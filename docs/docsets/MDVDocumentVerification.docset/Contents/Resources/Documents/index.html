Document Verification SDK for iOS

A fully automated way to verify that your users are signing up with genuine and physically present documents.

Document verification is a fully automated process that works on a wide range of global documents with different security features. Both Android and iOS SDKs come with a tested UI, complete with instructions and error messages that help users perform a successful scan.

To confirm that the ID document is genuine and physically present at the time of verification, we ask the user to tilt the document in four random directions. This lets our solution carry out a number of checks, including:

  • Document liveness [BETA]
    • Analyzing the way in which the user is moving the document to determine if it’s with them at all times.
  • Security feature analysis
    • Analyzing visual elements on the document and making sure they adhere to the guidelines set by the issuing authority.
  • Document image consistency
    • Comparing multiple frames from the camera feed to confirm the document hasn’t changed midway through the process.
  • Data extraction
    • Read and parse virtually any identity document in the world, be it a Malaysian MyTentera or a Californian driver‚Äôs license.
  • Data matching
    • Checking whether the document has been tampered with by comparing data on each side for consistency and validating the MRZ check digit.

DocumentVerification is a part of family of SDKs developed by Microblink for optical text recognition, barcode scanning, ID document, payment card scanning and many others.

Requirements

SDK package contains DocumentVerification framework and one or more sample apps which demonstrate framework integration. The framework can be deployed on iOS 11.0 or later.

Quick Start

Getting started with DocumentVerification SDK

This Quick Start guide will get you up and performing verification as quickly as possible. All steps described in this guide are required for the integration.

This guide closely follows the DocumentVerification-Sample app in the Samples folder of this repository. We highly recommend you try to run the sample app. The sample app should compile and run on your device.

The source code of the sample app can be used as the reference during the integration.

1. Initial integration steps

Using CocoaPods

  • Download and install/update Cocopods version 1.10.0 or newer

3. Initiating the scanning process

To initiate the scanning process, first decide where in your app you want to add scanning functionality. Usually, users of the scanning library have a button which, when tapped, starts the scanning process. Initialization code is then placed in touch handler for that button. Here we’re listing the initialization code as it looks in a touch handler method.

Swift

class ViewController: UIViewController, MDVDocumentVerificationOverlayViewControllerDelegate  {

    var documentVerificationRecognizer : MDVDocumentVerificationRecognizer?
    var blinkIdCombinedRecognizer : MDVBlinkIdCombinedRecognizer?

    override func viewDidLoad() {
        super.viewDidLoad()
    }

    @IBAction func didTapScan(_ sender: AnyObject) {

        /** Create BlinkId Combined recognizer */
        blinkIdCombinedRecognizer = MDVBlinkIdCombinedRecognizer()

        /** Create Document Verification recognizer */
        documentVerificationRecognizer = { class_prefix }}DocumentVerificationRecognizer(recognizer: blinkIdCombinedRecognizer!)

        /** Create Document Verification overlay settings */
        let settings : MDVDocumentVerificationOverlaySettings = MDV DocumentVerificationOverlaySettings()

        /** Create your overlay view controller */
        let documentVerificationOverlayViewController = MDVDocumentVerificationOverlayViewController(settings: settings, recognizer: documentVerificationRecognizer, delegate: self)

        /** Create recognizer view controller with wanted overlay view controller */
        let recognizerRunneViewController : UIViewController = MDVViewControllerFactory.recognizerRunnerViewController(withOverlayViewController: documentVerificationOverlayViewController)

        /** Present the recognizer runner view controller. You can use other presentation methods as well (instead of presentViewController) */
        self.present(recognizerRunneViewController, animated: true, completion: nil)
    }
}

@property (nonatomic, strong) MDVBlinkIdCombinedRecognizer * blinkIdCombinedRecognizer;

@implementation ViewController

4. License key

A valid license key is required to initalize scanning. You can generate a free trial license key, after you register, at Microblink developer dashboard.

You can include the license key in your app by passing a string or a file with license key. Note that you need to set the license key before intializing scanning. Ideally in AppDelegate or viewDidLoad before initializing any recognizers.

License key as string

You can pass the license key as a string, the following way:

Swift

MDVMicroblinkSDK.shared().setLicenseKey("LICENSE-KEY", errorCallback: block)

Objective-C

[[MDVMicroblinkSDK sharedInstance] setLicenseKey:@"LICENSE-KEY" errorCallback:block];

License key as file

Or you can include the license key, with the code below. Please make sure that the file that contains the license key is included in your project and is copied during Copy Bundle Resources build phase.

Swift

MDVMicroblinkSDK.shared().setLicenseResource("license-key-file", withExtension: "txt", inSubdirectory: "directory-to-license-key", for: Bundle.main, errorCallback: block)

Objective-C

[[MDVMicroblinkSDK sharedInstance] setLicenseResource:@"license-key-file" withExtension:@"txt" inSubdirectory:@"" forBundle:[NSBundle mainBundle] errorCallback:block];

If the licence is invalid or expired then the methods above will throw an exception.

5. Registering for scanning events

In the previous step, you instantiated MDVDocumentVerificationOverlayViewController object with a delegate object. This object gets notified on certain events in scanning lifecycle. In this example we set it to self. The protocol which the delegate has to implement is MDVDocumentVerificationOverlayViewControllerDelegate protocol. It is necessary to conform to that protocol. We will discuss more about protocols in Advanced integration section. You can use the following default implementation of the protocol to get you started.

Swift

func documentVerificationOverlayViewControllerDidFinishScanning(_ documentVerificationOverlayViewController: MDVDocumentVerificationOverlayViewController, state: MDVRecognizerResultState) {

    // this is done on background thread
    // check for valid state
    if state == .valid {

        // first, pause scanning until we process all the results
        documentVerificationOverlayViewController.recognizerRunnerViewController?.pauseScanning()

        DispatchQueue.main.async(execute: {() -> Void in
            // All UI interaction needs to be done on main thread
        })
    }
}

func documentVerificationOverlayViewControllerDidTapClose(_ documentVerificationOverlayViewController: MDVDocumentVerificationOverlayViewController) {
    // Your action on cancel
}

- (void)documentVerificationOverlayViewControllerDidFinishScanning:(MDVDocumentVerificationOverlayViewController *)documentVerificationOverlayViewController state:(MDVRecognizerResultState)state {

Built-in overlay view controllers and overlay subviews

Within DocumentVerification SDK there are several built-in overlay view controllers and scanning subview overlays that you can use to perform scanning.

Using MDVDocumentVerificationOverlayViewController

MDVDocumentVerificationOverlayViewController implements a new UI for scanning and verifying identity documents, which is optimally designed to be used with the DocumentVerificationRecognizer. It implements these features:

  • Onboarding screen showing instructions to hold the document in hand
  • Animations showing instructions how to tilt the document in four different directions
  • Visual indications when the user needs to place the document closer to the camera
  • Visual indications when the user needs to place the document farther from the camera
  • Help icon, which gets highlighted when users tap the screen
  • Additional onboarding available when the user presses the help icon
  • Timeout dialogue when scanning takes too long, with a call to action that opens additional onboarding
  • Clear indication to flip the document to show the back side
  • Clear indication for a searching phase, before the document is presented
  • Clear progress indication, when busy with OCR and data extraction
  • Clear message when the document is not supported

It has MDVDocumentVerificationViewControllerDelegate delegate which can be used out-of-the-box to perform scanning using the default UI. Here is an example how to use and initialize MDVDocumentVerificationOverlayViewController:

Swift

/** Create your overlay view controller */
let documentVerificationOverlayViewController : MDVDocumentVerificationOverlayViewController = MDVDocumentVerificationOverlayViewController(settings: documentVerificationSettings, recognizer: documentVerificationRecognizer, delegate: self)

/** Create recognizer view controller with wanted overlay view controller */
let recognizerRunneViewController : UIViewController = MDVViewControllerFactory.recognizerRunnerViewController(withOverlayViewController: documentVerificationOverlayViewController)

/** Present the recognizer runner view controller. You can use other presentation methods as well (instead of presentViewController) */
self.present(recognizerRunneViewController, animated: true, completion: nil)

UIViewController<MDVRecognizerRunnerViewController>* recognizerRunnerViewController = [MDVViewControllerFactory recognizerRunnerViewControllerWithOverlayViewController: documentVerificationOverlayViewController];

Overlay View Controller is an abstract class for all overlay views.

It’s responsibility is to provide meaningful and useful interface for the user to interact with.

Typical actions which need to be allowed to the user are:

MDVRecognizer and available recognizers

The MDVRecognizer concept

The MDVRecognizer is the basic unit of processing within the SDK. Its main purpose is to process the image and extract meaningful information from it. As you will see later, the SDK has lots of different MDVRecognizer objects that have various purposes.

Each MDVRecognizer has a MDVRecognizerResult object, which contains the data that was extracted from the image. The MDVRecognizerResult object is a member of corresponding MDVRecognizer object its lifetime is bound to the lifetime of its parent MDVRecognizer object. If you need your MDVRecognizerResult object to outlive its parent MDVRecognizer object, you must make a copy of it by calling its method copy.

While MDVRecognizer object works, it changes its internal state and its result. The MDVRecognizer object’s MDVRecognizerResult always starts in Empty state. When corresponding MDVRecognizer object performs the recognition of given image, its MDVRecognizerResult can either stay in Empty state (in case MDVRecognizerfailed to perform recognition), move to Uncertain state (in case MDVRecognizer performed the recognition, but not all mandatory information was extracted) or move to Valid state (in case MDVRecognizer performed recognition and all mandatory information was successfully extracted from the image).

As soon as one MDVRecognizer object’s MDVRecognizerResult within MDVRecognizerCollection given to MDVRecognizerRunner or MDVRecognizerRunnerViewController changes to Valid state, the onScanningFinished callback will be invoked on same thread that performs the background processing and you will have the opportunity to inspect each of your MDVRecognizer objects’ MDVRecognizerResult to see which one has moved to Valid state.

As soon as onScanningFinished method ends, the MDVRecognizerRunnerViewController will continue processing new camera frames with same MDVRecognizer objects, unless paused. Continuation of processing or reset recognition will modify or reset all MDVRecognizer objects’s MDVRecognizerResult. When using built-in activities, as soon as onScanningFinished is invoked, built-in activity pauses the MDVRecognizerRunnerViewController and starts finishing the activity, while saving the MDVRecognizerCollection with active MDVRecognizer.

MDVRecognizerCollection concept

The MDVRecognizerCollection is is wrapper around MDVRecognizer objects that has array of MDVRecognizer objects that can be used to give MDVRecognizer objects to MDVRecognizerRunner or MDVRecognizerRunnerViewController for processing.

The MDVRecognizerCollection is always constructed with array [[MDVRecognizerCollection alloc] initWithRecognizers:recognizers] of MDVRecognizer objects that need to be prepared for recognition (i.e. their properties must be tweaked already).

The MDVRecognizerCollection manages a chain of MDVRecognizer objects within the recognition process. When a new image arrives, it is processed by the first MDVRecognizer in chain, then by the second and so on, iterating until a MDVRecognizer object’s MDVRecognizerResult changes its state to Valid or all of the MDVRecognizer objects in chain were invoked (none getting a Valid result state).

You cannot change the order of the MDVRecognizer objects within the chain - no matter the order in which you give MDVRecognizer objects to MDVRecognizerCollection, they are internally ordered in a way that provides best possible performance and accuracy. Also, in order for SDK to be able to order MDVRecognizer objects in recognition chain in a best way possible, it is not allowed to have multiple instances of MDVRecognizer objects of the same type within the chain. Attempting to do so will crash your application.

List of available recognizers

This section will give a list of all MDVRecognizer objects that are available within DocumentVerification SDK, their purpose and recommendations how they should be used to get best performance and user experience.

Frame Grabber Recognizer

The MDVFrameGrabberRecognizer is the simplest recognizer in SDK, as it does not perform any processing on the given image, instead it just returns that image back to its onFrameAvailable. Its result never changes state from empty.

This recognizer is best for easy capturing of camera frames with MDVRecognizerRunnerViewController. Note that MDVImage sent to onFrameAvailable are temporary and their internal buffers all valid only until the onFrameAvailable method is executing - as soon as method ends, all internal buffers of MDVImage object are disposed. If you need to store MDVImage object for later use, you must create a copy of it by calling copy.

Success Frame Grabber Recognizer

The MDVSuccessFrameGrabberRecognizer is a special MDVecognizer that wraps some other MDVRecognizer and impersonates it while processing the image. However, when the MDVRecognizer being impersonated changes its MDVRecognizerResult into Valid state, the MDVSuccessFrameGrabberRecognizer captures the image and saves it into its own MDVSuccessFrameGrabberRecognizerResult object.

Since MDVSuccessFrameGrabberRecognizer impersonates its slave MDVRecognizer object, it is not possible to give both concrete MDVRecognizer object and MDVSuccessFrameGrabberRecognizer that wraps it to same MDVRecognizerCollection - doing so will have the same result as if you have given two instances of same MDVRecognizer type to the MDVRecognizerCollection - it will crash your application.

This recognizer is best for use cases when you need to capture the exact image that was being processed by some other MDVRecognizer object at the time its MDVRecognizerResult became Valid. When that happens, MDVSuccessFrameGrabberRecognizer's MDVSuccessFrameGrabberRecognizerResult will also become Valid and will contain described image.

Document Verification recognizers

Document Verification Recognizer

Use the MDVDocumentVerificationRecognizer to extract data from documents, determine document liveness and get information about security features on the scanned document. To check data extraction capabilities, including matching data from front and back sides of documents, see the Use MDVBlinkIdCombinedRecognizer documentation.

MDVDocumentVerificationRecognizer works in three steps:

  1. Front side data extraction
  2. Four tilting steps, used for additional verification
  3. Back side data extraction

The result contains:

  • The document liveness score
  • Security feature analysis score
  • Location and analysis result of every security feature
  • Data extraction results.

See the list of supported documents here. Contact us for additional information about the security features we support.

BlinkID Combined Recognizer

Use MDVBlinkIdCombinedRecognizer for scanning both sides of the supported document. First, it scans and extracts data from the front, then scans and extracts data from the barcode on the back, and finally, combines results from both sides. The BlinkIDCombinedRecognizer also performs data matching and returns a flag if the extracted data captured from the front side matches the data from the barcode on the back. You can find the list of the currently supported documents here. We will continue expanding this recognizer by adding support for new document types in the future. Star this repo to stay updated.

The SDK is localized on following languages: Arabic, Chinese simplified, Chinese traditional, Croatian, Czech, Dutch, Filipino, French, German, Hebrew, Hungarian, Indonesian, Italian, Malay, Portuguese, Romanian, Slovak, Slovenian, Spanish, Thai, Vietnamese.

If you would like us to support additional languages or report incorrect translation, please contact us at help.microblink.com.

If you want to add additional languages yourself or change existing translations, you need to set customLocalizationFileName property on MDVMicroblinkApp object to your strings file name.

For example, let’s say that we want to change text “Scan the front side of a document” to “Scan the front side” in BlinkID sample project. This would be the steps:

  • Find the translation key in en.strings file inside DocumentVerification.framework
  • Add a new file MyTranslations.strings to the project by using “Strings File” template
  • With MyTranslations.string open, in File inspector tap “Localize…” button and select English
  • Add the translation key “blinkid_generic_message” and the value “Scan the front side” to MyTranslations.strings
  • Finally in AppDelegate.swift in method application(_:, didFinishLaunchingWithOptions:) add MDVMicroblinkApp.instance()?.customLocalizationFileName = "MyTranslations"

Frequently asked questions and known problems

Here is a list of frequently asked questions and solutions for them and also a list of known problems in the SDK and how to work around them.

In demo everything worked, but after switching to production license I get NSError with MDVMicroblinkSDKRecognizerErrorDomain and MDVRecognizerFailedToInitalize code as soon as I construct specific MDVRecognizer object

Each license key contains information about which features are allowed to use and which are not. This NSError indicates that your production license does not allow using of specific MDVRecognizer object. You should contact support to check if provided licence is OK and that it really contains all features that you have purchased.

I get NSError with MDVMicroblinkSDKRecognizerErrorDomain and MDVRecognizerFailedToInitalize code with trial license key

Whenever you construct any MDVRecognizer object or, a check whether license allows using that object will be performed. If license is not set prior constructing that object, you will get NSError with MDVMicroblinkSDKRecognizerErrorDomain and MDVRecognizerFailedToInitalize code. We recommend setting license as early as possible in your app.

In my didFinish callback I have the result inside my MDVRecognizer, but when scanning activity finishes, the result is gone

This usually happens when using MDVRecognizerRunnerViewController and forgetting to pause the MDVRecognizerRunnerViewController in your didFinish callback. Then, as soon as didFinish happens, the result is mutated or reset by additional processing that MDVRecognizer performs in the time between end of your didFinish callback and actual finishing of the scanning activity. For more information about statefulness of the MDVRecognizer objects, check this section.

Disable logging
